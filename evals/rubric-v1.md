# 精錬品質ルブリック v1

## 概要

IdeaForge の精錬プロンプト出力を評価する基準。5軸 × 5段階 = 25点満点。
Phase 0 クリア基準: 平均 20/25 以上（全テストアイデアの平均）

---

## 評価軸

### 1. 具体性 (Specificity)

出力が具体的で、行動可能な情報を含んでいるか。

| スコア | 基準 |
|--------|------|
| 5 | 全フィールドが具体的。数値・固有名詞・具体例を含む。そのまま行動計画に使える |
| 4 | ほぼ具体的。1-2箇所に「〜など」「〜系」の曖昧表現があるが許容範囲 |
| 3 | 半分は具体的だが、主要フィールドに抽象的な記述が残る |
| 2 | 多くが抽象的。「様々な」「多くの」「幅広い」が頻出 |
| 1 | テンプレ的。どのアイデアにも当てはまる汎用的な記述 |

### 2. 市場理解 (Market Understanding)

ターゲット・競合・市場トレンドの分析が的確か。

| スコア | 基準 |
|--------|------|
| 5 | 競合分析が正確（実在サービス名+正しい特徴）。ターゲットが具体的ペルソナ。トレンドに追い風・向かい風の両面あり |
| 4 | 競合・ターゲットは正確。トレンド分析にやや深みが足りない |
| 3 | 主要競合は挙げているが分析が浅い。ターゲットが広すぎる |
| 2 | 競合の特徴が不正確、またはメジャーな競合を見落としている |
| 1 | 「競合なし」と主張、またはターゲットが「全員」 |

### 3. 実現可能性 (Feasibility)

技術難易度・リソース見積もり・リスク評価が現実的か。

| スコア | 基準 |
|--------|------|
| 5 | 技術要素・必要リソースが具体的。リスクと対策が対になっている。タイムラインが現実的 |
| 4 | 主要な技術・リソースは正確。リスク評価にやや楽観的な部分あり |
| 3 | 技術的にはほぼ正しいが、リソースやタイムラインが曖昧 |
| 2 | 技術難易度を過小/過大評価。重要なリスクを見落としている |
| 1 | 技術的に不正確な記述がある。リスク評価が欠如 |

### 4. 独自性 (Originality)

テンプレ的でない、アイデア固有の洞察・分析を含んでいるか。

| スコア | 基準 |
|--------|------|
| 5 | アイデア固有の洞察が3つ以上。「このアイデアだからこそ」の分析がある |
| 4 | 固有の洞察が2つ。一部はアイデアの文脈に合わせた分析 |
| 3 | 固有の洞察が1つ。残りは業界一般の分析 |
| 2 | ほぼ業界テンプレ。アイデアを入れ替えても成立する記述が多い |
| 1 | 完全にテンプレ。入力アイデアを無視しても生成できる内容 |

### 5. 読みやすさ (Readability)

構成・文章が明確で、意思決定に使えるか。

| スコア | 基準 |
|--------|------|
| 5 | 論理構造が明確。要約→詳細の流れ。数値にはコンテキスト付き。5分で意思決定可能 |
| 4 | 構造は明確。一部冗長な箇所があるが理解に支障なし |
| 3 | 読めるが、情報の整理が不十分。重要な情報が埋もれている |
| 2 | 冗長または断片的。読み返さないと理解できない |
| 1 | 構造がなく、情報の羅列。意思決定に使えない |

---

## 追加チェック項目（合否判定）

Step 1 (構造化) 固有:
- [ ] モード判定（A/B）が入力の情報量に対して正しいか
- [ ] title をユーザーに質問していないか
- [ ] target_users を入力外に勝手に拡大していないか
- [ ] differentiators が3個あるか

Step 2 (市場分析) 固有:
- [ ] TAM にハルシネーション数値がないか（出典なしの具体的数値は NG）
- [ ] 「競合なし」と主張していないか
- [ ] target_segments が Step 1 の target_users の範囲内か
- [ ] トレンドに追い風・向かい風の両方があるか
